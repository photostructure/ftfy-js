# TPP: Port roadmap (read me first)

## Summary

The index for the python-ftfy → `@photostructure/ftfy` port. It defines the order across the
other TPPs, what can run in parallel, and the hard serialization points. Start here, pick an
unblocked TPP, do the work in that file, and move it to `_done/` when finished. This file is
the map, not implementation work — keep the statuses below current as TPPs land.

## TPP maturity

- **Scoped (ready to implement now):** Wave 0 and Wave 1 TPPs have full task breakdowns.
- **Placeholder (`status: placeholder` in frontmatter):** Wave 2–4 TPPs are intentionally
  light per [docs/TPP-GUIDE.md](../docs/TPP-GUIDE.md) ("Placeholder TPPs"). Their first
  `Current phase` item is **"Research & scoping"** — a `/tpp` run on one should _scope it_
  (using the required reading and lore already provided) before writing code. Each carries
  enough context — dependencies, upstream files, tests, pitfalls — that scoping needs no
  re-planning.

## The waves

Each TPP names what it waits on ("after:"). A TPP is unblocked once everything in its "after"
note is done. Files are `_todo/20260529-<name>.md`.

### Wave 0 — foundation (scoped; start immediately)

Both can run at once with separate owners, and both must finish before any Wave 1 leaf that
consumes generated tables.

- **scaffolding-and-packaging** — after: nothing.
- **codegen-and-generated-tables** — after: nothing.

### Wave 1 — the parallel front (scoped; up to 5 at once after Wave 0)

The widest fan-out. **Start `strict-utf8-decoder` first** — it is the highest-risk module and
gates the critical path.

- **strict-utf8-decoder** ★ — after: scaffolding only.
- **config-and-types** — after: scaffolding only.
- **formatting-and-wcwidth** — after: scaffolding only (owns `gen_wcwidth.py`).
- **codecs-leaves** — after: codegen (charmaps). Leaves only; the dispatcher is Wave 2.
- **html-unescape** — after: codegen (html5-entities).

### Wave 2 — depends on specific Wave 1 work (placeholder)

`chardata` and `badness` are independent of each other and of the codec work, so all of these
can proceed in parallel once their inputs exist.

- **utf8-variants** — after: strict-utf8-decoder.
- **chardata** — after: codegen and html-unescape.
- **badness** — after: codegen (mojibake-categories).
- **codecs-dispatcher** — after: codecs-leaves, strict-utf8-decoder, and utf8-variants (it
  waits for all codec engines so it is never half-implemented).

### Wave 3 — orchestration (placeholder)

- **fixes** — after: chardata, badness, codecs-dispatcher, config-and-types.
- **index-orchestration** — after: fixes (and the rest of Wave 2).

`fixes` and `index-orchestration` share the `fixes ⇄ index` import cycle and are best done by
one owner, or a tightly-coordinated pair, together.

### Wave 4 — surface & verification (placeholder)

- **cli** — after: index-orchestration.
- **integration-tests-and-verification** — after: everything.

Tests travel with the code: each TPP ports its own subsystem's upstream tests (listed in its
required reading). The integration TPP owns only the cross-cutting suites — the JSON corpus,
doctests, CLI integration, and the release/drift gates.

## Critical path

The longest serial chain, which sets the minimum schedule, is:

codegen → strict-utf8-decoder → utf8-variants → codecs-dispatcher → fixes / index-orchestration
→ cli → integration-tests-and-verification.

Keep this chain moving; everything else slots around it in parallel.

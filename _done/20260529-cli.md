## Decisions & divergences

- **`os.EOL` translation.** Python writes through text-mode streams that translate
  `\n`→`os.linesep`. Mirrored via `translateEol` on every write (no-op on POSIX,
  CRLF on Windows). `CORRECT_OUTPUT` in the test uses Node's `os.EOL` to match.
- **`TextFixerConfig` over `makeConfig`.** cli.py constructs `TextFixerConfig(...)`;
  the port uses the public `TextFixerConfig` callable (not the module-internal
  `makeConfig`), keeping the CLI on the public surface.
- **Encode-error path is structurally present but practically unreachable.** Node
  always writes UTF-8 to stdout/files, so the `UnicodeEncodeError` branch can't
  fire; it's wired via an `OutputEncodeError` sentinel around the writes so the
  unix/windows messages and the cli.py control flow are faithfully reproduced and
  not dead code. No test exercises it (upstream has none either).
- **Usage/`-h` text.** Not pinned by `test_cli.py`; a reasonable argparse-style
  `USAGE`/`HELP_TEXT` is provided. Usage errors exit 2 (argparse convention).
- **`normalization` cast.** User input is passed through as `NormalizationForm`
  unchecked, like Python (an unsupported form errors later in `.normalize()`).

## Post-review fixes (codex review, vetted against `python -m ftfy.cli`, 2026-06-10)

Accepted and fixed:

- **Stale-dist hazard**: `tests/cli.test.ts` now rebuilds `dist/` unconditionally in
  `beforeAll` (previously only when missing, so edits to `src/cli.ts` could be silently
  untested).
- **Boolean long options reject `=value`**: `--guess=false` /
  `--preserve-entities=x` now exit 2 with argparse's
  `argument -g/--guess: ignored explicit argument 'false'` (verified verbatim).
- **Value-taking options refuse option-shaped values**: `-e --` and `-e -g` now exit 2
  with `expected one argument` like argparse; a bare `-` is still a legal value.
- **Short-flag clustering**: `-ge latin-1` == `-g -e latin-1` (argparse semantics),
  including `-h` inside a cluster.
- **Exact exit codes pinned**: decode-error and same-file tests assert exit 1 (cli.py's
  `sys.exit(1)`), not merely nonzero; usage errors assert 2.

Vetoed (documented, deliberate):

- Byte-for-byte argparse `-h`/usage text (line wrapping, etc.) — upstream tests don't
  pin help text, and reproducing argparse's wrapping algorithm is CPython-internals
  mimicry the CLAUDE.md "faithful ≠ slavish" rule excludes. Exit codes match; the help
  content covers the same options.

## Reviewer notes (scrutinize against cli.py)

- `pyRepr` vs Python `%r` of the encoding name — only ever a simple ASCII codec
  name or `"None"` in practice; single-quote preference mirrored.
- `splitKeepEnds` line-splitting vs Python binary-file iteration (split on `0x0a`,
  keep terminator, final unterminated line kept).
- The decode-error `%r` rendering when `-g` is used (encoding is `null`) now emits
  bare `None` (matching Python `%r % None`), not `'None'`. In practice this is
  unreachable: `guess_bytes` falls back to `sloppy-windows-1252`, which always
  decodes, so `-g` never surfaces a `DecodeError` to the CLI — but the rendering is
  correct if it ever did. Upstream has no test for it.

## Environment / running the codegen

- The codegen imports `ftfy` from the sibling `../python-ftfy` checkout (override with
  `FTFY_PYTHON_SRC`). ftfy needs `wcwidth`, which is **not** installable system-wide here
  (PEP 668 / no `python3-venv`). It is vendored into `.pyenv/` at the repo root
  (`python3 -m pip install --target .pyenv wcwidth`) and `.pyenv/` is gitignored.
  `_gen_common._setup_sys_path()` puts `.pyenv` + `../python-ftfy` on `sys.path`.
- **CI note:** the no-diff check must reproduce this env (checkout python-ftfy at the pinned
  SHA, `pip install --target .pyenv wcwidth`, then `python3 scripts/gen_all.py` and assert a
  clean tree). The GitHub Actions wiring is not part of this TPP — note it for whoever owns CI.

## Lore (read before touching the generators or their consumers)

- **Determinism:** `ts_string_literal()` emits ASCII-only literals (`\xHH`/`\uHHHH`/`\u{…}`),
  so output is byte-stable regardless of editor/locale. Keys are sorted (charmaps/html5) or in
  fixed upstream dict order (clues/categories). Verified: two runs → identical md5.
- **Header** records generator, `python-ftfy 6.3.1`, upstream commit
  `74dd0452b48286a3770013b3a02755313bd5575e`, and `unicodedata.unidata_version 15.0.0`.
  **Unicode skew is real:** Node here is ICU 78.3 ≈ Unicode 16.0 vs codegen's 15.0.0. The
  name/category tables are frozen at 15.0.0 (from Python); the drift test belongs to the
  chardata/explain_unicode consumers (compare `process.versions.unicode`).

- **charmaps.ts representation:** each table is a **256-char string** (index = byte). Single-byte
  decodes are all BMP, so `table.charCodeAt(byte)` is exact. `REAL_DECODING_HOLES[enc]` lists
  unassigned bytes — **strict decode must throw there**; the char stored at a hole is the
  Latin-1 placeholder (only so the string stays length-256). Sloppy tables have no holes and
  force `0x1A → U+FFFD`. `windows-1252` holes = `[0x81,0x8d,0x8f,0x90,0x9d]` (the CLI
  `-e windows-1252` decode-error test depends on this). `charmap.ts` builds the **encode** map
  at runtime (iterate 0..255, last-write-wins); no encode tables are emitted.
  - REAL set: latin-1, iso-8859-2/3/6/7/8/11, macroman, cp437, cp874, windows-1250..1258.
  - SLOPPY set: all 24 `INCOMPLETE_ENCODINGS` members (`sloppy-windows-125x`, `sloppy-cp125x`
    [aliases, identical tables — emitted for direct lookup], `sloppy-iso-8859-{3,6,7,8,11}`,
    `sloppy-cp874`). Consumers should `normalize_encoding`-style lookup; cp125x ≡ windows-125x.

- **encoding-regexes.ts:** `ENCODING_REGEX_SOURCES[enc]` is the **verbatim** Python
  `ENCODING_REGEXES[enc].pattern` (runtime string == Python's, control chars and all). Compile
  with the `u` flag. The 0x80-0xFF charlist contains no `] \ ^` and uses `-` only inside
  intended ranges, so no extra escaping is needed (the prefix `\x00-\x19\x1b-\x7f` and the
  high chars form valid classes under `u`). **Pitfall for chardata.ts:** Python `$` also
  matches just before a trailing `\n`; JS `$` (no `m`) does not — handle that nuance when
  porting `possible_encoding` (DESIGN "possible_encoding anchoring"). Order: ascii first.

- **utf8-clues.ts / mojibake-categories.ts:** raw class-body strings with `\N{}` already
  resolved (Python resolves at import). Interpolate inside `[...]`; `BADNESS_RE` /
  `UTF8_DETECTOR_RE` are assembled **in TS** (string-interpolate the Python verbose template,
  strip comments/whitespace, `u` flag). The test confirms every body is a valid `u`-flag class.
  `c1` = the 3-char string `U+0080 "-" U+009F` (a range).

- **html5-entities.ts:** the **full** CPython `html.entities.html5` (both `name;` and legacy
  non-`;` forms; some values are 2 codepoints). `unescape()` needs the full dict for its
  longest-prefix-without-semicolon loop; `_build_html_entities` (the `&name;` filter +
  uppercase variants) is ported **at runtime** in chardata.ts — not here.

- **unicode-names.ts (the explain_unicode contract):** to keep the table small (1.3 MB, lazy
  `await import()`), only **explicitly-named** codepoints are stored (32943). Algorithmic names
  are reconstructed at runtime:
  - CJK/Tangut/Khitan/Nushu/compat ideographs: `prefix + cp.toString(16).toUpperCase()` for
    `cp` in an `ALGORITHMIC_NAME_RANGES` run. Ranges are **auto-discovered** (a name is
    algorithmic iff it ends in `-<HEX(cp)>`), so they can't drift from the Python unidata
    version. Runs only cover assigned codepoints, so no `Cn` check is needed when deriving.
  - Hangul syllables U+AC00..U+D7A3: `"HANGUL SYLLABLE " + L+V+T` from `HANGUL_JAMO_{L,V,T}` +
    `HANGUL_S_BASE`/`N_COUNT`/`T_COUNT`. See `hangulName()` in `tests/generated.test.ts` for
    the exact runtime formula to copy into explain_unicode.
  - `UNICODE_CATEGORIES` is `[start,end,cat]` runs covering the **entire** 0..0x10FFFF space
    (unassigned → `Cn`), contiguous and gap-free; binary-search it at runtime.
  - **Verified:** reconstructing the name for all 1,114,112 codepoints (Hangul algo + ranges +
    explicit table) reproduces `unicodedata.name()` with **zero** mismatches. So the names can
    never be wrong; a range/Hangul bug would only inflate the explicit table.

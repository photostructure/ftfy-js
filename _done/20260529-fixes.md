## Decisions & divergences (this implementation)

- **`fix_surrogates` lone-surrogate matching.** Python's second substitution
  `SURROGATE_RE.sub("�", text)` runs on codepoints, so the astral char produced by
  `convert_surrogate_pair` is _not_ re-matched. In JS that char is still a UTF-16 surrogate
  pair, so a naive `[\ud800-\udfff]/g` clobbers it. Fixed by matching only **lone** surrogates:
  `/[\ud800-\udbff](?![\udc00-\udfff])|(?<![\ud800-\udbff])[\udc00-\udfff]/g`. This was the
  root cause of two initial test failures (`test_surrogates`, `test_ohio_flag`). **Reviewer:
  scrutinize this regex against the Python lone-vs-paired semantics.**
- **Import cycle.** Broke `fixes ↔ index` the ESM way: top-level `import * as ftfyIndex from
"./index.js"`, and only _call_ `ftfyIndex.fix_encoding` lazily inside the
  `decode_inconsistent_utf8` callback (never at module-eval). Verified init order both ways.
- **`registerFixers` seam.** `fixes.ts` calls `registerFixers({...})` once at import; `index.ts`
  imports `fixes.ts` so registration runs before any `FIXERS` lookup. This changed one Wave-1
  config test (`looking up an unregistered known fixer throws`) — that empty-registry guard is
  unreachable once `index.ts`/`fixes.ts` are loaded, so the test was updated to assert the
  fixers ARE wired. The Proxy guard itself still throws for config-only consumers.
- **`decode_escapes` `\N{...}`.** The named-character escape is intentionally unsupported (it
  needs the full Unicode-name table, ftfy never runs `decode_escapes` automatically, and no
  upstream test pins it). It throws a clear error; `\x`/`\u`/`\U`/octal/single-char all work.
- **`translate()` ports** (`remove_control_chars`, `fix_latin_ligatures`, `fix_character_width`)
  iterate by codepoint over the codepoint-keyed maps, `null` → drop.
- Byte fixers (`restore_byte_a0`, `replace_lossy_sequences`) are `(bin: string) => string` over
  binary strings; `ALTERED_UTF8_RE`/`A_GRAVE_WORD_RE` cloned with `g` and **no `u`**.

## Post-review fixes (codex review, vetted by the orchestrator, 2026-06-10)

- **`apply_plan` 'normalize' extension reverted.** The port had accepted a `normalize`
  plan step "so fix_and_explain plans round-trip"; verified against 6.3.1 that Python's
  `apply_plan` raises `ValueError("Unknown plan step: normalize")` and that ZERO corpus
  cases generate a normalize step (in Python or this port). The extension was innovation,
  not parity — `apply_plan` now throws, pinned by a test.
- **`fix_text` infinite loop fixed.** When `max_decode_length` landed inside a surrogate
  pair, the boundary was backed DOWN one unit, producing an empty segment and no forward
  progress (`fix_text("😀", null, {max_decode_length: 1})` hung). The boundary now rounds
  UP to include the whole pair — Python counts codepoints and can never split one.
  Regression tests added.
- **No-explanation branch mirrors the upstream quirk.** `fix_and_explain` with
  `steps === null` now calls `fix_encoding(text)` WITHOUT the config, exactly like
  `ftfy/__init__.py` — so encoding-fix sub-options are ignored when `explain` is off.
  Deliberately quirky; documented inline.
- **`decode_escapes` `\N{...}` stays unsupported** (clear throw): the upstream doctest
  only pins `\u`/`\x` escapes; supporting `\N` would need a sync name→codepoint table.
  Flagged to Matthew as a known, documented gap.

## Lore

- `fix_surrogates` regexes use **no `u` flag**; `convert_surrogate_pair` uses `charCodeAt`.
- `remove_control_chars` / width / ligature fixers use `translate`-by-codepoint, not
  code-unit `.replace`.
- `decode_inconsistent_utf8` calls back into `index.ts` — import the **namespace** lazily,
  never destructure at top level (breaks the cycle).
- Byte fixers are typed `(bin: string) => string`; never feed binary strings to Unicode helpers.
